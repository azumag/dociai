# OBS・配信表示モード (issue #14)

## 方針: 操作用UIと表示用UIの分離

- `index.html` — 操作卓。配信者だけが見る。配信画面には映さない。
- `obs.html` — 表示専用。最新コメント・AI応答・発話中ペルソナ (ON AIRランプ) のみを表示し、操作要素を置かない。

2つのページは同一オリジンの `BroadcastChannel("dociai-obs")` で接続され、操作卓が
`comment` / `reply` / `speech` イベントを送る。

## 使い方

```bash
python3 scripts/serve.py 8080
```

1. `http://localhost:8080/` を開いて操作卓を使う
2. **同じブラウザ**で `http://localhost:8080/obs.html` を開く
3. OBS側は次のどちらかで取り込む

### 方法A: ウィンドウキャプチャ (現PoCの推奨)

`obs.html` を開いたブラウザウィンドウをOBSの「ウィンドウキャプチャ」で取り込む。
BroadcastChannel は**同じブラウザ内でしか届かない**ため、この方法が確実。

### 方法B: OBSブラウザソース (制約あり)

OBSブラウザソースに `http://localhost:8080/obs.html?transparent=1` を指定すると
透明背景で読み込める。ただしOBS内蔵ブラウザは操作卓とは別プロセスのため、
**現状のBroadcastChannel実装ではデータが届かない**。表示レイアウトの確認用。

将来対応: ローカル常駐アプリ化の際に小さなWebSocketサーバー (例: `ws://localhost:8765`) を
立て、BroadcastChannelの代わりに配信する。`obs.js` の受信部を差し替えるだけで済む構造にしてある。

## 透明背景モード

`obs.html?transparent=1` で `body` 背景が透過になる。各表示要素は半透明の
ダークカードなので、明るいゲーム画面の上でも読める。

## 音声出力先の扱い

読み上げ (Web Speech API) は**操作卓 (index.html) 側**で再生される。OBSに音声を乗せる方法:

| 方法 | 手順 |
|---|---|
| デスクトップ音声 (簡単) | OBSの「デスクトップ音声」にブラウザの音を含める |
| 仮想オーディオデバイス (推奨) | macOS: BlackHole / Windows: VB-Cable にブラウザの出力を向け、OBSでそのデバイスをキャプチャ。配信者のマイクと分離できる |

表示が邪魔になりすぎないよう、`obs.html` は画面下部に最大720px幅で寄せてある。
OBS側でクロップ・拡縮して調整する。

## マイク監視とのフィードバックループ (issue #32)

マイク監視 (`micMonitor`) はマイク入力の音量から配信者の発話を検知し、発話中はAI音声
キューを保留する。スピーカーで音声を再生している環境 (ヘッドホン不使用) では、AI自身の
声がスピーカーからマイクに回り込み、自分の発話を誤って「配信者が話した」と検知して
自己中断してしまう可能性がある。

- `getUserMedia` の `echoCancellation`/`noiseSuppression` を既定で有効にしており、
  同一タブ内での音声のループバックはブラウザの標準的なエコーキャンセルである程度
  抑制される。
- それでも誤検知する場合は、しきい値 (`micMonitor.threshold`) を上げるか、上表の
  **仮想オーディオデバイス** (BlackHole / VB-Cable) にAI音声を向け、配信者のマイクと
  音響的に分離するのが最も確実。ヘッドホンで音声を聞く運用でも同様に回避できる。
- ElectronはWebのようなマイク選択UIを持たないため、OSのデフォルト入力デバイスが
  誤って仮想デバイスや別マイクになっていると気づきにくい。`micMonitor.deviceId`
  (設定パネルの「マイク監視」タブの device欄) で監視対象を明示的に固定できる。
